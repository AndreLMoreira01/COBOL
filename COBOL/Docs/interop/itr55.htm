5.4 Calling COBOL from .NET

There are three ways to call and interact with an ACUCOBOL-GT program from a .NET assembly. You can:

  • Use ACUCOBOL-GT's .NET compiler options to generate Microsoft Intermediate Language (MSIL) code that can call and run your ACUCOBOL-GT program. .NET programmers can then invoke these objects as they would any native .NET resource.

    • Though perhaps easiest, this option limits data passing as discussed in section 5.4.1.

  • Use the .NET native interface, "wrunnet.dll", to interact with the COBOL program at the API level. "wrunnet.dll" contains a singleton class, CVM, that encapsulates the ACUCOBOL-GT runtime. With the CVM, the .NET programmer can programmatically instantiate the ACUCOBOL-GT runtime and invoke a COBOL program without a COM interface or knowledge of the Windows API or .NET PINVOKE.

    • .NET programmers can include "wrunnet.dll" in a Visual Studio .NET project and take advantage of the native .NET development environment. This interface gives them lower-level control over the COBOL program, and is very similar to the ACUCOBOL-GT C and Java interfaces.

  • Use the .NET proxy, "Interop.AcuGTObjects.dll", in conjunction with the ACUCOBOL-GT COM server.

    • The COM server is a COM object containing the ACUCOBOL-GT Windows runtime DLL. The proxy provides all interface and data marshalling between .NET and the COM server. (Marshalling is the process of gathering data and transforming it into a standard format before transmitting it over a network.)
