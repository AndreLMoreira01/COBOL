3.4.2.2 Calling ACUCOBOL-GT Vision data from ODBC applications

AcuODBC is designed to let Windows users read and write ACUCOBOL-GT Vision files as well as relative files. Because none of these file formats is relational, AcuODBC's job is to receive ODBC calls from ODBC-enabled applications (like Microsoft Word) and translate them into the appropriate I/O actions.

After AcuODBC translates the ODBC calls into Vision I/O, it processes these requests for data and returns results to the driver manager, which passes them along to the calling Windows application. In this case, the ODBC components work together in this environment as follows:

Application

The Windows application generates an SQL query to accomplish a file I/O task.

Driver manager

The driver manager receives calls from the Windows application, establishes which data source driver is being used, and calls AcuODBC.

Driver

AcuODBC receives SQL commands from the driver manager and translates them from SQL to Vision or relative file I/O.

Data sources

The data source is accessed by AcuODBC. The database manager performs the necessary I/O operation and passes the results back to AcuODBC. If your data resides on a different computer, you may access these files locally or remotely using one of the following services:

    • AcuODBC Server performs SQL processing remotely on an application server, and returns the results to the client machine. This service is described in the AcuODBC User's Guide.
    • AcuServer® locates files on a remote server and returns them to the local machine for processing. This service is described in the AcuServer User's Guide.
